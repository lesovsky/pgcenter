# Task 04: Pre-deploy QA

## Required Skills

Перед выполнением задачи загрузи:

## Description

Финальная приёмочная задача фичи горизонтального скролла таблицы статистики (issue #14). Задачи 01–03 уже реализованы (чистая функция окна колонок + состояние `scrollOffset`, рендеринг замороженной колонки и видимого окна, горячие клавиши `[`/`]` со сбросом offset и help-текстом). Цель этой задачи — убедиться, что вся фича собрана, проходит полный набор проверок и удовлетворяет всем критериям приёмки из user-spec и tech-spec перед мержем в master и деплоем.

Это QA-задача без написания кода: прогоняем автоматические гейты (`make test`, `make lint`, `make vuln`), затем сверяем каждый критерий приёмки. Часть критериев (визуальное поведение скролла, заморозки, маркеров) проверяется только вручную в узком терминале — автоматически их утвердить нельзя, поэтому по ним собираем подтверждение пользователя на наборе stat-экранов (activity, tables, pg_stat_io).

Найденные дефекты не чиним в этой задаче — фиксируем как findings и возвращаем на доработку в соответствующую кодовую задачу.

## What to do

1. Прочитать источники истины: user-spec («Критерии приёмки», «Как проверить»), tech-spec («Acceptance Criteria», «Agent Verification Plan»), и decisions-log (отклонения от плана, если есть).
2. Phase 1 (окружение): pgcenter — Go-проект без docker-compose dev-окружения. Убедиться, что собирается свежий код из текущей ветки: `make build`. Если сборка падает — это critical-finding, остановить QA и сообщить ошибку.
3. Phase 2 (автоматические гейты): прогнать полный набор и зафиксировать результат каждого:
   - `make test` — race detector + coverage, все тесты `top` зелёные, без регрессий.
   - `make lint` — golangci-lint + gosec без замечаний.
   - `make vuln` — govulncheck без уязвимостей.
4. Phase 3 (сверка автоматических критериев): по результатам тестов подтвердить unit-покрытие `visibleColumns`, сброс offset в обоих путях переключения view, клампинг `[`/`]`, отсутствие регрессий в существующих тестах `top`.
5. Phase 4 (ручная проверка): дать пользователю чёткий чек-лист ручных проверок в узком терминале (≈100 символов) на наборе экранов activity, tables, pg_stat_io и собрать его подтверждение по каждому пункту (скролл `[`/`]`, заморозка первой колонки, маркеры `‹`/`›`, сброс offset при переключении включая `S` → procpidstat, no-op на широком терминале, bold имени замороженной колонки, поведение на пустом результате).
6. Свести итог: для каждого критерия приёмки — PASS / FAIL / нужно ручное подтверждение пользователя. Перечислить все findings со severity. Дать вердикт о готовности к деплою.

## Acceptance Criteria

Полный свод критериев из user-spec и tech-spec, который эта задача обязана проверить.

**Автоматические гейты:**
- [ ] `make build` — сборка проходит без ошибок.
- [ ] `make test` — все тесты зелёные, без регрессий (`Test_switchViewTo`, `Test_orderKeyLeft/Right`, `Test_alignViewToResult` и др.).
- [ ] `make lint` — golangci-lint + gosec без замечаний.
- [ ] `make vuln` — govulncheck без уязвимостей.

**Критерии из tech-spec (проверяются по коду/тестам):**
- [ ] `visibleColumns` имеет unit-тесты: all-fit, scroll-right, mid-scroll, past-end clamp, very-narrow, оба флага маркеров.
- [ ] `config.scrollOffset` сбрасывается в 0 в `viewSwitchHandler` И в `switchViewToProcPidStat`.
- [ ] `[`/`]` зарегистрированы на view `sysstat`; клампятся на обоих краях.
- [ ] Первая колонка всегда рендерится независимо от offset; offset индексирует только колонки `1..Ncols-1`.
- [ ] Маркеры `‹`/`›` рендерятся только когда есть скрытые колонки в соответствующую сторону.
- [ ] Help-экран документирует `[`/`]`.

**Критерии из user-spec (часть — ручная проверка пользователя):**
- [ ] `]` сдвигает окно на одну колонку вправо, `[` — влево; первая колонка всегда видна. *(ручная)*
- [ ] На границах скролл не уходит за пределы. *(ручная + unit)*
- [ ] Таблица помещается целиком → `[`/`]` no-op, маркеры не отображаются. *(ручная)*
- [ ] При переключении на другой экран offset сбрасывается в 0, включая переход на procpidstat через `S`. *(ручная + unit)*
- [ ] `‹` тогда и только тогда, когда есть скрытое слева; `›` — когда есть скрытое справа. *(ручная)*
- [ ] Offset сохраняется между авто-обновлениями в пределах одного экрана. *(ручная)*
- [ ] Сортировка (`←`/`→`) и скролл (`[`/`]`) ортогональны. *(ручная + unit)*
- [ ] Имя замороженной первой колонки выводится жирным. *(ручная)*
- [ ] Скролл работает на всех stat-экранах основной таблицы; extra-панели (iostat/netdev/fsstats/logtail) не затронуты. *(ручная)*
- [ ] Заморозка + сортировка на первой колонке сосуществуют без визуального конфликта. *(ручная)*
- [ ] Пустой результат (0 строк): заголовок со скроллом и маркерами корректен, без сбоя. *(ручная)*

## Verification Steps

**Автоматические:**
- `make build` → бинарь собирается без ошибок.
- `make test` → весь набор зелёный, без регрессий в пакете `top`.
- `make lint` → нет замечаний golangci-lint / gosec.
- `make vuln` → нет уязвимостей govulncheck.
- Сводный гейт фичи: `make test && make lint && make vuln` — всё зелёное.

**Ручные (подтверждает пользователь, в терминале ≈100 символов):**
- Экран **activity** (`a`): `query` не помещается, виден маркер `›`; нажать `]` несколько раз — окно скроллится, `pid` остаётся на месте, появляется `query` и слева маркер `‹`; нажать `[` обратно к нулю — `‹` исчезает.
- Экран **tables** (`t`): скролл `[`/`]` работает, первая колонка (имя таблицы) заморожена; имя первой колонки в заголовке — жирное.
- Экран **pg_stat_io** (`j`): скролл по дальним счётчикам, первая колонка остаётся; под-экраны count/time не сломаны.
- **Сброс offset:** проскроллить вправо, переключиться на другой экран и обратно — offset = 0; отдельно проверить переход на **procpidstat** через клавишу `S` — offset тоже сброшен.
- **No-op на широком терминале** (≈200 символов): открыть экран, где всё помещается (например databases `d`) — `]` ничего не делает, маркеры `‹`/`›` отсутствуют.
- **Extra-панели** (iostat/netdev/fsstats/logtail) — не затронуты скроллом.

## Details

**Тип задачи:** приёмочное тестирование (QA), кода не пишем. Дефекты только фиксируем как findings и возвращаем в кодовые задачи 01–03, не чиним здесь.

**Окружение:** pgcenter — Go-проект, без docker-compose dev-окружения. Phase 1 «Environment Rebuild» из скилла сводится к `make build` на свежем коде текущей ветки. Те же гейты выполняет CI (`default.yml` / `release.yml`) в контейнере `lesovsky/pgcenter-testing:0.0.10` — локальный прогон должен соответствовать им.

**Команды:**
- `make build` — сборка в `./bin/pgcenter`.
- `make test` — race detector + coverage.
- `make lint` — golangci-lint + gosec.
- `make vuln` — govulncheck.

**Граница «агент / пользователь»:** визуальное поведение TUI (скролл, заморозка, маркеры, bold, no-op, ресайз) автоматизированного харнесса не имеет — эти критерии закрываются ТОЛЬКО ручным подтверждением пользователя. Агент готовит точный чек-лист и собирает ответ; не помечать визуальные критерии как PASS без явного подтверждения пользователя.

**Edge cases для ручной проверки:**
- Очень узкий терминал — помещается только первая колонка: деградация без паники.
- Ресайз терминала — окно пересчитывается, offset зажимается.
- Пустой результат (0 строк) — заголовок со скроллом/маркерами работает, данных нет.
- Первая колонка одновременно заморожена и является колонкой сортировки — подсветки сосуществуют (по Decision 4 приоритет у сортировки).

**Implementation hints:**
- Сверять критерии построчно: для каждого — статус PASS / FAIL / WAITING-USER + источник доказательства (тест/лог/подтверждение).
- Если `make lint`/`make vuln` падают на коде вне фичи — отметить как pre-existing, не блокировать фичу, но указать в findings.
- Severity findings: critical (сборка/тесты падают, критерий не выполнен), major (регрессия, частично невыполненный критерий), minor (косметика).
- При FAIL — указать конкретный критерий, ожидание vs факт, и в какую задачу (01/02/03) возвращать.
