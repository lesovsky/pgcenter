# User Spec: record/report для экранов 0.11.0 (bgwriter, replslots, pg_stat_io, JIT)

## Что делаем

Добавляем поддержку `pgcenter record` и `pgcenter report` для четырёх экранов, появившихся в
релизе 0.11.0 и пока работающих только в интерактивном TUI (помечены как незаписываемые):
bgwriter (`pg_stat_bgwriter` + `pg_stat_checkpointer`), replslots (`pg_replication_slots` ⟕
`pg_stat_replication_slots`), pg_stat_io (два под-экрана: count и time) и statements_jit
(JIT-метрики `pg_stat_statements`). После фичи `pgcenter record` будет собирать их статистику
в архив, а `pgcenter report` — проигрывать её офлайн, как уже умеет для tables/activity/wal/
statements_*.

## Зачем

Сейчас эти четыре экрана видны только «вживую» в TUI. Их данные нельзя сохранить для
последующего разбора. DBA, расследующий инцидент постфактум или анализирующий тренды, теряет:
частоту и тип контрольных точек (bgwriter/checkpointer), рост удержанного WAL по слотам
репликации (replslots — частая причина переполнения диска), разбивку IO по типам
бэкендов/объектов/контекстов (pg_stat_io) и стоимость JIT-компиляции (statements_jit).
Роадмап 0.11.0 сознательно отложил record/report как TUI-first срез (ADR
[004-feat-bgwriter-checkpointer]); эта фича закрывает отложенный пробел.

## Пользовательские истории

- Как DBA, я хочу, чтобы `pgcenter record` записывал статистику bgwriter/replslots/pg_stat_io/
  JIT в архив, чтобы анализировать её офлайн после инцидента.
- Как DBA, я хочу проигрывать каждый новый экран через `pgcenter report`, чтобы видеть те же
  поинтервальные дельты, что и в TUI, но из сохранённого архива.
- Как DBA, я хочу видеть описание колонок нового отчёта через `pgcenter report -d`, чтобы
  интерпретировать незнакомый экран.

### Пользовательские сценарии

**Сценарий 1: запись новых статистик**
1. DBA запускает `pgcenter record -f stats.tar` в окне инцидента.
2. Рекордер собирает все пригодные к записи вьюхи, включая четыре новых.
3. В архиве `stats.tar` появляются entry `bgwriter.<TS>.json`, `replslots.<TS>.json`,
   `stat_io.<TS>.json`, `stat_io_time.<TS>.json`, `statements_jit.<TS>.json` рядом с прежними.
4. Результат: статистика новых экранов сохранена для офлайн-разбора.

**Сценарий 2: проигрывание отчёта**
1. DBA запускает `pgcenter report -f stats.tar -B` (bgwriter).
2. Система читает архив, считает дельты между соседними снимками и печатает отчёт в том же
   формате, что и существующие отчёты (заголовок со временем, периодический заголовок колонок).
3. DBA повторяет для других экранов: `-L` (replslots), `-J c` / `-J t` (pg_stat_io count/time),
   `-X j` (JIT).
4. Результат: те же данные, что показывал TUI, но из архива.

**Сценарий 3: описание отчёта**
1. DBA запускает `pgcenter report -d -B`.
2. Система печатает описание колонок отчёта bgwriter.
3. Результат: DBA понимает значение каждой колонки.

## Дизайн и интерфейс

Изменения только в CLI `pgcenter report`; TUI не меняется. Формат вывода отчётов идентичен
существующим экранам report.

### Команды (флаги `pgcenter report`)

- **`-B` / `--bgwriter`** (bool) — отчёт bgwriter (`pg_stat_bgwriter` / `pg_stat_checkpointer`).
- **`-L` / `--replslots`** (bool) — отчёт replslots (`pg_replication_slots` /
  `pg_stat_replication_slots`).
- **`-J` / `--io`** (string, значения `c`/`t`) — отчёт pg_stat_io: `c` → count (`stat_io`),
  `t` → time (`stat_io_time`). Идиома повторяет `-X`/`-D`/`-P` (строковый флаг с под-значением).
- **`-X` / `--statements`** расширяется значением **`j`** → JIT (`statements_jit`). Отдельный
  флаг не нужен — JIT это под-экран pg_stat_statements, как в TUI под циклом `x`.
- **`-d` / `--describe`** работает с каждым из новых флагов и печатает описание колонок.

Короткие буквы `B`, `L`, `J` проверены — свободны.

### Формат вывода

Идентичен существующим отчётам: строка с таймштампом и rate, периодический заголовок колонок
(каждые 20 строк), значения по колонкам. Набор и названия колонок совпадают с
соответствующим TUI-экраном. Семантика дельт повторяет TUI: кумулятивные счётчики
вычитаются между снимками; абсолютные и текстовые колонки (например, возраст статистики у
bgwriter, статус WAL у replslots, ключ строки у pg_stat_io) проходят как есть.

### UX-поведение

- **Пустые данные** (нет слотов у replslots; `jit_functions=0` у JIT; старый архив без новых
  entry) → печатается только заголовок, без строк. Спец-сообщений INFO/WARNING нет (это
  обычные SQL-вьюхи, поведение как у tables/wal).
- **Неизвестное значение `-J`/`-X`** → ошибка выбора типа отчёта, как у существующих
  строковых флагов.

## Как должно работать

### Основной сценарий

1. `pgcenter record` начинает собирать четыре экрана по тому же SQL-пути, что и существующие
   записываемые вьюхи (как wal/tables), складывая статистику каждого типа в архив за каждый тик.
2. `pgcenter report -<флаг>` читает архив, по версии PostgreSQL из метаданных выбирает раскладку
   экрана, считает дельты между соседними снимками и печатает отчёт.

### Граничные случаи

- **Remote-режим:** ограничений нет — это чистые SQL-экраны (в отличие от procpidstat), они
  доступны и при удалённом подключении; записываются как есть.
- **Старый архив** (записан до фичи): нет данных новых типов → отчёт печатает только заголовок.
  Обратная совместимость сохраняется, версия формата архива не меняется.
- **Смена версии PG внутри архива:** при изменении версии в метаданных отчёт заново выбирает
  раскладку; version-aware экраны (bgwriter/stat_io/statements_jit) переключаются автоматически.
- **JIT без pg_stat_statements:** как и все statements-экраны, JIT не записывается, если
  расширение недоступно; `report -X j` по такому архиву → только заголовок.
- **Нулевые значения в дельта-колонках при проигрывании:** записанные данные уже не содержат
  NULL в дельта-колонках (нулевые значения подставлены на этапе записи), поэтому подсчёт дельт
  при проигрывании не падает.

## Критерии приёмки

- [ ] `pgcenter record` записывает в архив статистику всех четырёх экранов (5 типов отчёта:
      bgwriter, replslots, stat_io, stat_io_time, statements_jit) — entry каждого типа за тик.
- [ ] Флаги report работают: `-B`→bgwriter, `-L`→replslots, `-J c`→stat_io, `-J t`→stat_io_time,
      `-X j`→statements_jit; неизвестные значения `-J`/`-X` дают ошибку, как у существующих.
- [ ] `report -d <флаг>` печатает описание колонок для каждого из 5 типов.
- [ ] Проигранные отчёты показывают корректные дельты: кумулятивные счётчики вычитаются,
      абсолютные/текстовые колонки проходят без изменений; значения совпадают с тем, что
      показывает TUI на тех же данных.
- [ ] Пустой/старый архив печатает только заголовок, без падения и лишних INFO/WARNING.
- [ ] Проигрывание корректно на PG14–18 — раскладка version-aware экранов соответствует
      версии записи.
- [ ] Выплачен техдолг [007] (поведенческая проверка диффа pg_stat_io на нулевых ячейках) и
      [004] (устранение дублирования констант индексов колонок procpidstat между пакетами).
- [ ] Обновлены доки: `overview.md`, `architecture.md`, features-catalog (перестают называть
      эти экраны TUI-only/незаписываемыми).
- [ ] Все проверки качества зелёные: `make test` / `make lint` / `make vuln` и CI на матрице
      PG14–18.

## Ограничения

- Это чисто SQL-экраны: механизм записи и формат хранения архива не меняются (в отличие от
  procpidstat — никакого procfs-обогащения и stateful-рекордера).
- Инфраструктура проигрывания уже готова: существующий report-пайплайн умеет собирать раскладку
  version-aware экранов по версии записи; рабочий прецедент — экран `replication`, который
  записывается и проигрывается тем же путём.
- Нельзя ломать существующие записываемые экраны и связанные с ними проверки.
- Go 1.25.11; новых зависимостей нет; все гейты зелёные на PG14–18.
- Поддерживаемые версии по экранам: bgwriter/replslots — PG14+, statements_jit — PG15+,
  stat_io/stat_io_time — PG16+ (версионный гейт записи отрабатывает после снятия запрета на
  запись).

## Риски

- **Риск 1:** при проигрывании отчёт перестраивает SQL экрана, имея на руках только версию PG
  (без полного набора параметров записи). **Митигация:** этот SQL в report не исполняется —
  проигрываются записанные данные, из перестроенной конфигурации берётся лишь раскладка экрана;
  построение шаблона на неполных параметрах не падает. Покрыто golden-тестами.
- **Риск 2:** нулевые/пустые значения в дельта-колонках при проигрывании. **Митигация:**
  нулевые значения подставлены ещё на этапе записи, в архиве NULL отсутствует.
- **Риск 3:** при включении записи этих экранов нужно поправить счётчики в тестах фильтрации, и
  они отличаются по версиям PG. **Митигация:** точечное обновление + гейт CI на матрице PG14–18.
- **Риск 4:** коллизия коротких букв CLI. **Митигация:** `B`/`L`/`J` проверены — свободны.

## Технические решения

- Мы решили **не менять рекордер и формат хранения** — для чистых SQL-экранов достаточно
  включить их запись; общий SQL-путь сбора и report-пайплайн уже всё умеют.
- Мы решили **переиспользовать флаг `-X` для JIT** (значение `j`), а не вводить отдельный —
  JIT это под-экран pg_stat_statements, как в TUI.
- Мы решили **дать pg_stat_io один строковый флаг `-J c|t`** на два типа отчёта, а не два
  bool-флага — идиома `-X`/`-D`/`-P`.
- Мы решили **не делать remote-гейт** — эти вьюхи доступны и удалённо (в отличие от procpidstat).
- Мы решили **не добавлять спец-обработку «нет данных»** — пусто=пусто, как у tables/wal.
- Мы решили **проверять дельты golden-файлами на специально собранном архиве-входе**, а не
  перегенерировать общий легаси-фикстур (он не содержит данных новых экранов, записан на ~PG13,
  и его перегенерация сломала бы все 30 существующих golden-эталонов).
- Мы решили **обновить доки внутри фичи** — иначе они врут до релиза.
- Мы решили **выплатить техдолг [007] и [004]** (оба на пути фичи), а [006] и [005] оставить
  долгом (нужен живой standby / не связано с record/report).

## Тестирование

**Unit-тесты:** делаются всегда. Обновление счётчиков в тестах фильтрации записываемых вьюх и
флагов их записываемости; маппинг новых CLI-флагов в типы отчётов; наличие описаний для всех
пяти типов.

**Интеграционные тесты:** делаем — проверка диффов через golden-файлы для пяти типов отчётов,
с входными данными из специально собранного архива (не легаси-фикстура). Для version-aware
экранов — golden на версиях 14/17/18, чтобы подтвердить переключение раскладки. Сторона записи
покрыта существующей проверкой состава записываемых экранов. Плюс поведенческая проверка
подсчёта дельт на нулевых ячейках (техдолг [007]).

**E2E тесты:** не делаем отдельный живой record→report end-to-end — такого харнесса в проекте
нет, его постройка несоразмерна; собранный архив + golden дают ту же diff-проверку
версионно-параметрично. Шов «живая запись ↔ живое проигрывание» закрывается ручной проверкой
(см. ниже).

## Как проверить

### Агент проверяет

| Шаг | Инструмент | Ожидаемый результат |
|-----|-----------|-------------------|
| 1. `make test` | bash | Все тесты зелёные, включая обновлённые тесты фильтрации, новые golden-тесты отчётов и поведенческую проверку дельт |
| 2. `make lint` | bash | golangci-lint + gosec без замечаний |
| 3. `make vuln` | bash | govulncheck чистый |
| 4. `go test ./report/...` | go test | Вывод отчётов по 5 типам совпадает с golden-эталонами, включая версионные варианты |
| 5. `go test ./cmd/report/... ./record/...` | go test | Маппинг новых флагов в типы отчётов и состав записываемых экранов соответствуют ожиданиям |

### Пользователь проверяет

- Записать живой архив: `pgcenter record -f /tmp/s.tar` на работающем PG, затем
  `pgcenter report -f /tmp/s.tar -B`, `-L`, `-J c`, `-J t`, `-X j` — убедиться, что отчёты
  печатаются с корректными дельтами и колонками.
- **Закрыть шов «запись ↔ проигрывание»:** хотя бы для одного кумулятивного экрана (например
  bgwriter или stat_io) сравнить значения в живом отчёте с тем, что показывает TUI на тех же
  данных — убедиться, что дельты сходятся.
- Проверить `pgcenter report -d -B` (и для остальных флагов) — описание колонок печатается.
- Проверить старый архив без новых данных — отчёт печатает только заголовок, без ошибок.
