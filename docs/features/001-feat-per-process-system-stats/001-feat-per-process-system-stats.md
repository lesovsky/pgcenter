# User Spec: Per-process System Stats Screen

## Что делаем

Добавляем новый TUI-экран "Per-process system stats" в `pgcenter top`, открываемый по `Shift+S`.
Экран совмещает данные из `pg_stat_activity` и procfs (`/proc/[pid]/stat`, `/proc/[pid]/io`)
и показывает для каждого PostgreSQL-бэкенда утилизацию CPU и IO в реальном времени.
Closes issues #118 (CPU per backend) и #123 (IO per backend).

## Зачем

DBA при troubleshooting видит долгоживущий запрос в `pg_stat_activity`, но не может определить,
является ли он CPU-bound или IO-bound без переключения на внешние инструменты (`top`, `pidstat`, `iotop`).
Существующая статистика блочных устройств (`Shift+B`) показывает агрегированный IO по дискам,
без привязки к конкретным процессам. Новый экран позволяет соотнести SQL-активность
с системными ресурсами непосредственно в pgcenter, не покидая инструмент.

## Пользовательские истории

- Как DBA, я хочу видеть утилизацию CPU каждого PostgreSQL-бэкенда в реальном времени, чтобы быстро найти процесс, создающий CPU-нагрузку, без переключения на внешние инструменты.
- Как DBA, я хочу видеть IO-активность каждого бэкенда (чтение и запись в KiB/s), чтобы отличить CPU-bound запросы от IO-bound.
- Как DBA, я хочу видеть накопленные метрики CPU и IO с момента старта бэкенда, чтобы оценить суммарный вклад процесса в нагрузку за всё время его жизни.
- Как DBA, я хочу фильтровать idle-процессы, чтобы сосредоточиться только на активных бэкендах.
- Как DBA, я хочу фильтровать бэкенды по возрасту, чтобы видеть только долгоживущие процессы.

### Пользовательские сценарии

**Сценарий 1: Поиск CPU-виновника**
1. DBA замечает высокую нагрузку на сервер PostgreSQL.
2. DBA открывает `pgcenter top` и нажимает `Shift+S`.
3. Система открывает экран "Per-process system stats" с таблицей всех бэкендов.
4. DBA нажимает `←`/`→` для смены колонки сортировки, выбирает `%all`.
5. DBA видит бэкенд с `%all=95`, `%us=94`, `%sy=1` — запрос CPU-bound.
6. DBA смотрит колонку `query` — видит текст тяжёлого запроса прямо в той же строке.
7. Результат: DBA определил виновника CPU без выхода из pgcenter.

**Сценарий 2: Диагностика IO-bound запроса**
1. DBA видит деградацию производительности, CPU сервера невысок.
2. DBA открывает экран `Shift+S`, сортирует по `read,KiB/s`.
3. DBA видит процесс с `read,KiB/s=450`, `%all=12` — запрос IO-bound, CPU почти не использует.
4. DBA смотрит `wait_event` — видит `DataFileRead`, подтверждение IO-ожидания.
5. Результат: подтверждён IO-bound запрос с конкретными IO-метриками и wait event.

**Сценарий 3: Анализ с фильтрацией**
1. На системе 200+ idle-соединений (connection pooler), мешают анализу.
2. DBA нажимает `I` — idle-бэкенды скрываются, остаются только активные.
3. DBA нажимает `A`, вводит `30` (секунды) — остаются только бэкенды старше 30 секунд.
4. Результат: отфильтрованный список с действительно подозрительными процессами.

**Сценарий 4: Анализ параллельного запроса**
1. DBA видит несколько строк с одинаковым `datname`/`usename` но разными `pid`.
2. Это параллельные воркеры одного запроса — каждый виден отдельной строкой.
3. DBA суммирует `%all` воркеров — понимает реальную загрузку параллельного плана.

**Сценарий 5: Запуск без прав на /proc/[pid]/io**
1. DBA запускает pgcenter от имени своего пользователя (не `postgres`, не `root`).
2. При открытии экрана `Shift+S` в строке статуса появляется warning:
   "Cannot read /proc/[pid]/io: permission denied. Run as postgres user or via sudo."
3. CPU-колонки (`%all`, `%us`, `%sy`, `all_total,s`, `us_total,s`, `sy_total,s`) отображаются нормально.
4. IO-колонки (`read,KiB/s`, `write,KiB/s`, `read_total,KiB`, `write_total,KiB`) пустые для всех строк.
5. Результат: экран частично работоспособен, пользователь знает о причине.

## Дизайн и интерфейс

### Экран "Per-process system stats"

Открывается по `Shift+S`, занимает главную область TUI (как экраны `activity`, `tables` и др.).
Строка-заголовок с агрегированной системной статистикой (CPU, память и т.д.) отображается
всегда — стандартное поведение pgcenter, не меняется.

### Колонки (17 штук, в порядке отображения)

Соглашение об именах: `name,unit` — накопленные; `%name` — CPU rate в %; `name,KiB/s` — IO rate.

| # | Имя колонки      | Источник               | Формат   | Описание                                              |
|---|------------------|------------------------|----------|-------------------------------------------------------|
| 1 | pid              | pg_stat_activity       | integer  | PID бэкенда                                           |
| 2 | datname          | pg_stat_activity       | text     | База данных                                           |
| 3 | usename          | pg_stat_activity       | text     | Пользователь                                          |
| 4 | state            | pg_stat_activity       | text     | Состояние (active, idle, idle in transaction, …)      |
| 5 | wait_etype       | pg_stat_activity       | text     | Тип ожидания (wait_event_type, сокращение)            |
| 6 | wait_event       | pg_stat_activity       | text     | Событие ожидания                                      |
| 7 | all_total,s      | /proc/[pid]/stat       | HH:MM:SS | Accumulated CPU: utime+stime с момента старта         |
| 8 | us_total,s       | /proc/[pid]/stat       | HH:MM:SS | Accumulated user-mode CPU time                        |
| 9 | sy_total,s       | /proc/[pid]/stat       | HH:MM:SS | Accumulated system-mode CPU time                      |
|10 | read_total,KiB   | /proc/[pid]/io         | KiB      | Accumulated bytes read (syscall level)                |
|11 | write_total,KiB  | /proc/[pid]/io         | KiB      | Accumulated bytes written (syscall level)             |
|12 | %all             | /proc/[pid]/stat       | %        | CPU rate: (Δutime+Δstime)/Δtime, нормализован 0–100% |
|13 | %us              | /proc/[pid]/stat       | %        | User CPU rate, нормализован 0–100%                    |
|14 | %sy              | /proc/[pid]/stat       | %        | System CPU rate, нормализован 0–100%                  |
|15 | read,KiB/s       | /proc/[pid]/io         | KiB/s    | IO read rate                                          |
|16 | write,KiB/s      | /proc/[pid]/io         | KiB/s    | IO write rate                                         |
|17 | query            | pg_stat_activity       | text     | Текст текущего запроса (обрезается по ширине)         |

Примечание по IO-семантике: `read_bytes`/`write_bytes` из `/proc/[pid]/io` считают байты,
прошедшие через read/write syscalls — они включают page cache и не равны фактическому
дисковому IO. Это стандартная семантика `pidstat -d`.

### UX-поведение

- **Навигация по колонкам и сортировка:** стандартные контролы pgcenter:
  `←`/`→` — выбор колонки сортировки; `↑`/`↓` — изменение ширины текущей колонки;
  `<` — переключение направления сортировки (desc/asc); `/` — установка текстового фильтра.
  По умолчанию — сортировка по `pid` по возрастанию.
- **Фильтр `I`:** переключает скрытие бэкендов с `state = 'idle'`.
  Уже реализован глобально в pgcenter как `toggleIdleConns` — новый экран наследует автоматически.
- **Фильтр `A`:** открывает диалог ввода порога возраста в секундах.
  Скрывает бэкенды, у которых `now() - query_start < порог`.
  Уже реализован глобально как `dialogChangeAge` — новый экран наследует автоматически.
- **Обновление:** интервал и механизм обновления — стандартные для pgcenter (хоткей `z`).
- **Исчезновение процесса:** если бэкенд завершился между тиками, его строка
  пропадает на следующем обновлении (как на activity-экране).
- **Недоступность `/proc/[pid]/io`:** при первом открытии экрана `Shift+S` pgcenter
  пробует прочитать `/proc/self/io`; если получен `EACCES` — выводится warning один раз
  за сессию в строке статуса. IO-колонки (`read,KiB/s`, `write,KiB/s`, `read_total,KiB`, `write_total,KiB`)
  пустые для всех строк на протяжении всей сессии. CPU-колонки работают штатно.
  Если `/proc/[pid]/io` конкретного PID недоступен в момент чтения (race condition,
  отличный пользователь) — IO-значения этой строки пустые, остальные строки не затрагиваются.
- **Remote mode:** экран недоступен при удалённом подключении.
  При попытке открыть — предупреждение: "Per-process stats available in local mode only".

## Как должно работать

### Основной сценарий (local mode)

1. Пользователь нажимает `Shift+S` в `pgcenter top`.
2. pgcenter проверяет, что подключение локальное (`db.Local == true`).
3. pgcenter проверяет доступность `/proc/[pid]/io` попыткой прочитать `/proc/self/io`;
   при `EACCES` — выводит warning один раз за сессию, IO-метрики не собираются.
4. Система запрашивает `pg_stat_activity` — получает список бэкендов с PID.
5. Для каждого PID читает:
   - `/proc/[pid]/stat` → `utime`, `stime` (jiffies)
   - `/proc/[pid]/io` → `read_bytes`, `write_bytes` (если доступно)
6. Вычисляет дельту относительно предыдущего тика:
   - На первом тике rate-колонки (`%all`, `%us`, `%sy`, `read,KiB/s`, `write,KiB/s`) отображаются как 0;
     дельта начинает вычисляться со второго тика.
   - Rate-метрики: `%all = (Δutime + Δstime) / Δtime / cpu_count × 100`; `read,KiB/s`/`write,KiB/s` аналогично.
   - Accumulated-метрики: `utime + stime` → `all_total,s` (HH:MM:SS); `read_bytes` → `read_total,KiB`.
7. Объединяет данные pg_stat_activity и procfs по `pid`, формирует строки таблицы.
8. Отображает таблицу в главной области, применяет активные фильтры.

### Граничные случаи

- Если `/proc/[pid]/stat` недоступен для PID (процесс завершился в момент чтения) — строка пропускается, исчезнет на следующем тике без паники.
- Если `/proc/self/io` недоступен при открытии экрана (`EACCES`) — warning один раз за сессию, IO-колонки пустые для всех строк, CPU-метрики продолжают работать. Если `/proc/[pid]/io` отдельного PID недоступен (race condition) — IO этой строки пустое, остальные не затрагиваются.
- Remote mode — экран не активируется, показывается предупреждение.
- Параллельные воркеры — каждый отображается отдельной строкой со своим PID.
- Бэкенды с `state = 'idle'` — отображаются, пока не активирован фильтр `I`.

## Ограничения

- **Только local mode.** procfs (`/proc/[pid]/*`) физически недоступен при удалённом мониторинге. Remote monitoring через PL/Perl для этого экрана не реализуется.
- **Требования к правам для IO-метрик.** `/proc/[pid]/io` доступен только если pgcenter запущен от того же пользователя, что и PostgreSQL-бэкенды (обычно `postgres`), через `sudo`, или с `CAP_SYS_PTRACE`. При отсутствии прав IO-колонки пустые, warning при открытии экрана.
- **Aux-процессы PostgreSQL не отображаются.** Checkpointer, bgwriter, WAL writer, archiver отсутствуют в `pg_stat_activity` и не попадают на экран.
- **IO-метрики — syscall level, не дисковый IO.** `/proc/[pid]/io` считает байты через read/write syscalls, включая page cache. Реальный дисковый IO может быть меньше.
- **Нет per-process iowait.** Метрика iowait на уровне отдельного процесса недоступна из procfs; для её получения требуется Netlink taskstats API. Это вынесено за рамки данной версии.
- **Нет горизонтального скроллинга.** Все 17 колонок должны помещаться в ширину терминала (минимум 200 символов для отображения без усечения числовых колонок). На более узких терминалах колонка `query` усекается первой.
- **Record/report не поддерживается в v1.** Рекордер pgcenter работает только через SQL-запросы; интеграция procfs-данных в подсистему записи вынесена в отдельный issue для следующей версии.

## Риски

- **Риск: `/proc/[pid]/io` недоступен по умолчанию при запуске не от пользователя `postgres`.** Это стандартное поведение Linux (ptrace_scope=1). **Митигация:** graceful degradation (empty IO-колонки + warning), документирование требования к правам в `pgcenter --help` и в README.
- **Риск: 17 колонок могут не влезть на узких терминалах (<200 символов).** **Митигация:** тест на разных размерах; при необходимости — уменьшение минимальной ширины числовых колонок.
- **Риск: несоответствие IO-метрик ожиданиям DBA** (ожидает дисковый IO, видит syscall). **Митигация:** документирование семантики в help-экране pgcenter.

## Технические решения

- Мы решили реализовать экран только для local mode, потому что procfs физически недоступен при remote monitoring, а реализация через PL/Perl выходит за рамки scope.
- Мы решили не показывать aux-процессы (checkpointer и др.), потому что они отсутствуют в `pg_stat_activity` — единственном доступном источнике сопоставления PID с SQL-активностью.
- Мы решили нормализовать CPU% в диапазон 0–100%, потому что этот диапазон интуитивнее для DBA, чем N×100% при multi-core.
- Мы решили использовать паттерн accumulated + rate (как на statements-экране), потому что это даёт полную картину: rate отвечает на "кто нагружает сейчас", accumulated — на "кто за жизнь потратил больше всего ресурсов".
- Мы решили показывать каждый параллельный воркер отдельной строкой, а не агрегировать под лидера, потому что это проще в реализации и позволяет видеть PID каждого воркера.
- Мы решили использовать `HH:MM:SS` для accumulated CPU-времени, потому что этот формат привычен DBA (аналог вывода `ps`).
- Мы решили сократить `wait_event_type` до `wait_etype` в заголовке колонки для экономии ширины.
- Мы решили вынести iodelay (per-process iowait) за рамки v1, потому что его реализация требует Netlink taskstats API (нет в codebase), что значительно увеличило бы scope. Планируется в отдельном issue.
- Мы решили не включать колонку `cancelled_write_bytes` (ccwr), потому что она слабо востребована при troubleshooting и экономит ширину экрана.
- Мы решили исключить record/report из v1, потому что рекордер работает только через SQL-запросы к `view.Views`; интеграция procfs-данных потребовала бы расширения архитектуры рекордера и значительно увеличила бы scope.

## Тестирование

**Unit-тесты:** делаются всегда, не обсуждаются.

**Интеграционные тесты:** делаем — сбор procfs-данных с реальными PID (PID самого тестового процесса), вычисление rate/accumulated метрик на двух тиках, корректная обработка `EACCES` при недоступности `/proc/[pid]/io`.

**E2E тесты:** не делаем — TUI не поддаётся автоматизированному тестированию агентом.

### Пользователь проверяет

- Запустить `pgcenter top` (от пользователя `postgres` или через `sudo`), нажать `Shift+S` — открылся экран "Per-process system stats" с 17 колонками.
- Запустить тяжёлый CPU-запрос (например, рекурсивный CTE или full-scan большой таблицы), найти его PID в экране, сравнить `%all` визуально с выводом `top -p <pid>`.
- Аналогично проверить `read,KiB/s`/`write,KiB/s` с `pidstat -d 1 -p <pid>`.
- Нажать `I` — idle-бэкенды исчезли; нажать `I` снова — появились.
- Нажать `A`, ввести `10` — показываются только бэкенды старше 10 секунд.
- Запустить pgcenter от непривилегированного пользователя, нажать `Shift+S` — появился warning об отсутствии прав на IO, CPU-колонки работают, IO-колонки пустые.
- Подключиться к remote PostgreSQL, нажать `Shift+S` — появилось предупреждение об ограничении local mode.
