# User Spec: Обзорный дашборд состояния инстанса

## Что делаем

Добавляем в `pgcenter top` новый полноэкранный режим — обзорный дашборд состояния инстанса, открываемый горячей клавишей `v` (o**v**erview). В отличие от остальных экранов pgcenter (построчные таблицы статистики), это **карточный** экран: метрики сгруппированы в карточки вида «заголовок + строки label:value». Дашборд агрегирует ключевые сигналы здоровья инстанса из нескольких источников на одном экране и служит навигационным хабом — окинул взглядом, понял где болит, осознанно перешёл по хоткею в детальный экран.

Это первая версия (v1): фокус на каркасе и концепции — раскладке карточек, пути рендеринга, механике сбора (три внутренних ритма + backoff по латентности), деградации недоступных метрик в `n/a` и работе системных метрик на удалённом подключении. Точный итоговый состав карточек дорабатывается вторым заходом после оценки MVP.

## Зачем

У DBA, подключившегося к проблемному инстансу, в pgcenter нет экрана для оценки здоровья «одним взглядом». Чтобы собрать общую картину, приходится перебирать множество экранов (`activity`, `databases`, `bgwriter`, `replslots`, `pg_stat_io`, `statements`) и держать числа в голове. Прямой конкурент — pg_activity — имеет более богатый обзорный заголовок, но (а) он статичен и никуда не ведёт, (б) его системные метрики (CPU/mem/IO/load) работают только при локальном подключении.

Дашборд даёт один экран для ответа на вопрос «что вообще происходит с инстансом» и отправную точку для прицельного drill-down.

**Ключевой дифференциатор vs pg_activity.** Системные карточки (CPU/mem/swap/load) работают и на удалённом подключении — через уже существующую PL/Perl-схему `pgcenter`. У конкурента system stats по сети не работают в принципе. То есть это не клон обзорника pg_activity, а обзорник без его главного ограничения.

**Почему карточный экран, а не строка-заголовок.** Ценность хаба — в композиции: на одном кадре собраны сигналы со всех важных экранов, чтобы не переключаться в хаосе по хоткеям в стрессовой ситуации, а оценить состояние и перейти осознанно.

## Пользовательские истории

- Как дежурный DBA, я хочу по нажатию `v` за один взгляд оценить здоровье инстанса, чтобы нырнуть в самый проблемный экран одним осознанным нажатием.
- Как DBA на удалённом инстансе, я хочу чтобы системные карточки (CPU/mem/swap/load) работали по сети через PL/Perl, чтобы получить host-контекст, который pg_activity по сети показать не может.
- Как DBA, я хочу видеть `n/a` вместо `0` для недоступных метрик (выключен `track_io_timing`, нет `pg_stat_statements`, старая версия PG), чтобы не принять пропуск данных за реальный ноль.
- Как DBA на задыхающемся инстансе, я хочу чтобы обзорник не подсыпал лишней нагрузки на сервер, чтобы мониторинг не усугублял инцидент.

### Пользовательские сценарии

**Сценарий 1: триаж инцидента (ключевой — «обзорник поймал то, что иначе собирали бы руками»)**
1. В 03:00 срабатывает алерт по лагу реплики. DBA подключается к мастеру и нажимает `v`.
2. Карточка **Sessions** выглядит нормально, карточка **System** показывает низкий load — значит это не насыщение хоста.
3. При этом карточка **DB time** показывает database time, резко превышающий интервал тика (бэкенды жгут процессорное время), а на карточке **Checkpointer** чекпоинты идут как `requested`, а не `timed` — давление по WAL.
4. Результат: один взгляд исключил хост и указал на checkpoint/WAL-шторм. DBA нажимает `b`, чтобы подтвердить на экране bgwriter — ещё не открыв ни одного другого экрана. Без обзорника пришлось бы под давлением вручную обойти `activity` → `bgwriter` → `databases`, собирая ту же картину по кускам.

**Сценарий 2: host-контекст по сети**
1. DBA подключается к удалённому инстансу, на котором установлена PL/Perl-схема `pgcenter`, и нажимает `v`.
2. Карточка **System** показывает реальные load average, разбивку CPU, память и swap — данные хоста, прочитанные через PL/Perl.
3. Если схема `pgcenter` не установлена — карточка System показывает `n/a`, а в статусной строке (cmdline) выводится одно сообщение, поясняющее, что системные метрики недоступны без схемы.

**Сценарий 3: честный `n/a`**
1. На инстансе выключен `track_io_timing`; DBA открывает дашборд.
2. Метрики, зависящие от выключенной настройки, показывают литеральное `n/a` — не `0` и не пустоту.
3. DBA сразу понимает, что метрика недоступна по конфигурации, а не равна нулю.

## Дизайн и интерфейс

Интерфейс — CLI/TUI (gocui). Новых форм ввода нет.

### Экраны
- **Overview (хоткей `v`):** новый полноэкранный экран. Открывается по `v`, закрывается переходом на любой другой экран по его хоткею. **Не** является экраном по умолчанию — `top` по-прежнему стартует на `activity` (сохраняем мышечную память). Перерисовывается на каждом тике обновления, как живой экран.

### Ключевые компоненты — карточки v1
Карточка = заголовок группы + строки `label: value`. Состав v1 (reuse-first + database time; целевой набор шире — см. «Технические решения»):

- **Instance:** версия, uptime, recovery-статус, строка подключения.
- **Sessions:** total / active / idle / idle in txn / idle in txn aborted / waiting.
- **DB time:** database time (дельта `sum(total_exec_time)`) — сильный сигнал нагрузки; longest xact.
- **Checkpointer:** чекпоинты timed / requested, write / sync ms.
- **System (host):** load average, разбивка CPU, память, swap (вариант A — без IO/disk в v1).

Рядом с заголовком каждой группы показывается хоткей её детального экрана как подсказка для drill-down — например `Checkpointer [b]`, `Sessions [a]`. Карточки **не** интерактивны (нет выделения/Enter-перехода) — переход выполняется обычным нажатием уже знакомого хоткея.

### UX-поведение
- **Деградация в `n/a`.** Каждая метрика собирается независимо; при недоступности источника (нет прав, выключена настройка, версия PG без нужного view, нет расширения) ячейка показывает литеральное `n/a` — никогда `0` и никогда пусто.
- **Независимость карточек.** Падение или таймаут одного источника гасит в `n/a` только его ячейку; остальной дашборд продолжает рендериться. Это сильнее поведения табличных экранов, где ошибка сэмпла гасит всю таблицу.
- **Первый тик.** На первом тике (ещё нет предыдущей выборки) метрики-дельты показывают `n/a`, и в статусной строке (cmdline) выводится временная подсказка `collecting...`, которая гаснет после первого успешного рефреша. Так первый-тиковый `n/a` отличим от постоянно-недоступного.
- **Backoff медленного источника.** Когда внутренний backoff разредил опрос медленного источника, его карточка продолжает показывать последнее известное (stale) значение — не `n/a`. Backoff молчаливый (без сообщений в cmdline) и сам восстанавливается, когда латентность спадает.
- **Узкий терминал.** Карточки компактные и рассчитаны на размещение в один кадр на 80×24. Если терминал меньше минимального размера — показывается guard-сообщение «terminal too small». Сбрасывания карточек и вертикального скролла в v1 нет.
- **Интервал обновления.** Управляется единственной ручкой — существующим хоткеем `z`. Три внутренних ритма сбора (дёшево-свежее / реже-дорогое / медленно-кэшируемое) и backoff — это деталь реализации, второй пользовательской ручки не появляется.

## Как должно работать

### Основной сценарий
1. Пользователь нажимает `v`.
2. pgcenter переключается на экран overview, собирает метрики из источников и рендерит карточки.
3. На каждом тике обновления (интервал задаётся `z`) карточки перерисовываются с актуальными значениями и дельтами.
4. Пользователь оценивает состояние, видит подсказку-хоткей нужной группы и нажимает его — переходит в детальный экран.

### Граничные случаи
- **Remote без PL/Perl-схемы** → карточка System в `n/a` + одно cmdline-сообщение при переключении.
- **Standby / recovery** → WAL/checkpoint/lag используют recovery-aware пути (checkpoint→restartpoint, lag с точки зрения standby).
- **Нет `pg_stat_statements`** → метрики database time в `n/a`.
- **PG 14/15** (нет `pg_stat_io`), **< 17** (checkpointer внутри `pg_stat_bgwriter`) → значения, корректные для версии, либо `n/a`; без NULL-заполненных колонок.
- **Выключен `track_io_timing`** → зависящие метрики в `n/a`.
- **Первый тик** → дельты в `n/a` + `collecting...` в cmdline.
- **Нулевая геометрия после пейджера/редактора** (`maxX==0`) → рендер пропускается до восстановления размеров.

## Критерии приёмки
- [ ] Нажатие `v` открывает экран overview; `top` по умолчанию по-прежнему стартует на `activity`.
- [ ] Все карточки v1 умещаются в один кадр на терминале 80×24 без обрезки; при размере меньше минимального показывается guard-сообщение «terminal too small» (без скролла и без сбрасывания карточек).
- [ ] Каждая метрика обновляется на тике; метрики-дельты показывают корректную дельту; на первом тике дельты показывают `n/a`, а в cmdline — временную подсказку `collecting...`, которая гаснет после первого рефреша.
- [ ] Недоступные метрики показывают литеральное `n/a` (никогда `0`, никогда пусто).
- [ ] На удалённом подключении с PL/Perl-схемой `pgcenter` карточка System показывает реальные host-метрики (CPU/mem/swap/load — вариант A); без схемы — `n/a` и одно cmdline-сообщение (ровно один `printCmdline` на путь).
- [ ] Падение/таймаут одного источника гасит в `n/a` только его карточку; остальной дашборд рендерится (независимость карточек).
- [ ] Дорогие источники используют медленный ритм; backoff по латентности увеличивает интервал пропуска медленного источника; источник под backoff показывает последнее известное (stale) значение, а не `n/a`; второй пользовательской ручки интервала не появляется.
- [ ] Рядом с заголовком каждой группы отображается хоткей её детального экрана (например `Replication [o]`).
- [ ] Версионно/конфигурационно-зависимые ячейки деградируют в `n/a` или версионно-корректные значения на PG 14/15, < 17, при выключенном `track_io_timing`, без `pg_stat_statements`; на standby WAL/checkpoint/lag используют recovery-aware пути.

## Ограничения
- **Версии PostgreSQL:** активная поддержка PG 14–18.
- **Стек:** Go 1.25+, gocui (TUI), pgx/v5.
- **Вариант A в v1:** карточка System использует уже собираемые каждый тик CPU/mem/swap/load; host IO/disk **не входят** в v1 (отложены в v1.1). Никакой переделки взаимоисключающего `collectExtra`-механизма и новой ветки сбора disk/fs в v1 нет.
- **Путь рендеринга:** переиспользуем существующую free-form `Fprintf`-идиому (прецедент `printSysstat`/`printPgstat`/`help`), а не табличный `PGresult`-рендерер.
- **TUI-only в v1:** view регистрируется с `NotRecordable: true` (прецедент ADR [004]/[008]); поддержка `record`/`report` отложена в backlog (запись композитного экрана агрегатов концептуально мутная — пересмотрим отдельно, если будет запрос).
- **Единственная ручка интервала:** `z`; три ритма и backoff — деталь реализации.

## Риски
- **Риск (узкий терминал):** карточки v1 могут не уместиться на маленьком терминале. **Митигация:** компактный дизайн + min-size guard; property-тест доказывает инвариант «умещается-или-guard».
- **Риск (стоимость нового SQL на нагруженном инстансе):** database time требует нового агрегата `sum(total_exec_time)`, выполняемого на мониторимом инстансе. **Митигация:** три ритма + backoff по латентности; минимальный набор «дёшево-свежих» источников; кэширование на медленном ритме.
- **Риск (single-table предположение в рендер-цикле):** `printStat` сейчас безусловно рисует `dbstat` каждый тик; ветвление по имени view инвазивно для общего цикла. **Митигация:** минимальное условное создание/рендер по образцу панели `extra`; табличный путь для остальных экранов не трогаем; ветку покрываем юнит-тестом.
- **Риск (повторный живой пользователь `NotRecordable`):** после ADR [008] ни один production-view не выставлял этот флаг. **Митигация:** низкий риск — поле и ветка `filterViews` сохранены; нужно лишь обновить count-тесты.

*Примечание: риск дестабилизации боковых панелей (iostat/nicstat/fsstats) через always-on сбор disk/fs устранён решением о варианте A — самой инвазивной ветки в v1 нет.*

## Технические решения
- Мы решили **переиспользовать существующую free-form `Fprintf`-идиому** (прецедент `printSysstat`/`help`) с новым gocui-view и принтером `printOverview`, потому что путь рендеринга карточек label:value уже фактически есть в коде; табличный `renderDbstat` привязан к `PGresult` и для карточек не подходит, а собственный card-движок — переусложнение.
- Мы решили **не делать карточки интерактивными в v1** (вместо Enter-перехода — подсказка-хоткей рядом с заголовком), потому что free-form gocui-view не имеет модели выделения ячеек, а курсор + подсветка + маппинг карточка→экран — это большой объём новой TUI-механики ради маргинальной выгоды. Навигабельные карточки — в backlog.
- Мы решили **взять вариант A** (карточка System = CPU/mem/swap/load), потому что дифференциатор «remote system stats через PL/Perl» полностью обеспечивается им, эти метрики и так собираются каждый тик, и это убирает единственную инвазивную переделку фичи. Host IO/disk — в v1.1 (drill-down туда — одно нажатие `j`/`B`/`F`).
- Мы решили **спрятать tiering сбора как деталь реализации** (новая константа `CollectOverview` по образцу хука `CollectExtra` из ADR [001]; три ритма + backoff живут внутри собственной collect-функции на `Collector`), потому что пользователю нужна одна ручка `z`, а не три.
- Мы решили **брать xid age как `max(age(datfrozenxid))` по `pg_database`** (дёшево, по одной строке на БД), потому что per-relation возраст — это drill-down, а не обзорная метрика.
- Мы решили **показывать `collecting...` в cmdline на первом тике** и **держать stale-значение при backoff**, потому что так первый-тиковый `n/a` отличим от постоянного, а разреженный опрос — это «реже обновляем», а не «данные недоступны».

## Тестирование

**Unit-тесты:** делаются всегда, не обсуждаются. Покрытие: чистая layout-функция раскладки карточек с property-тестом по размерам терминала (идиома `visibleColumns` из 009, инвариант «умещается-или-guard»); вывод `printOverview` против `bytes.Buffer`; форматирование `n/a` (sentinel → литерал `n/a`, отличимый от реального `0`); гейтинг трёх ритмов (счётчик тиков) и backoff (мок медленного источника → удержание stale-значения) — без живого PG.

**Интеграционные тесты:** делаем — против PG 14–18: сбор метрик по источникам и пути graceful degradation (нет расширения, старая версия, выключен `track_io_timing`, remote без схемы).

**E2E тесты:** не делаем автоматически — в проекте нет автоматизированного TUI E2E-харнесса. Проверка ручная (см. ниже).

### Пользователь проверяет

- Проверить remote-подключение с PL/Perl-схемой и без неё — System показывает данные либо `n/a` + cmdline-сообщение.
- Проверить на standby и на инстансе с выключенным `track_io_timing` / без `pg_stat_statements` — соответствующие метрики в `n/a`, дашборд не падает.
- Понаблюдать первый тик (`collecting...` в cmdline, гаснет) и поведение при искусственно медленном источнике (карточка держит stale-значение, без спама в cmdline).
