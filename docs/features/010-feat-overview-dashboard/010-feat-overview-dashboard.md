# User Spec: Verbose-режим верхних панелей (расширенный обзор инстанса)

## Что делаем

Добавляем в `pgcenter top` режим **verbose** для двух верхних сводных панелей — `sysstat` (слева) и `pgstat` (справа), — переключаемый горячей клавишей `v`. В обычном (compact) режиме панели остаются как сейчас — по 4 строки. В verbose-режиме:

- левая панель `sysstat` прирастает на **3 строки**: одностроковые агрегаты `iostat`, `nicstat`, `filesyst`;
- правая панель `pgstat` прирастает на **5 строк**: `workload`, `databases`, `workers`, `replication`, `bgwr/ckpt`.

Режим — это **переключатель поверх текущего экрана** (как сайд-панели `B`/`F`/`N`/`L` или idle-фильтр `I`), а не отдельный экран. Verbose **персистентен** между экранами: включил — расширенная сводка здоровья едет с тобой и остаётся над любой детальной таблицей (`activity`, `tables`, …). Это не card-grid и не новый full-screen view — это дополнительные строки label:value в уже существующих панелях.

## Зачем

Две верхние панели pgcenter — это, по сути, **уже готовый мини-обзор инстанса** (и системные метрики в `sysstat` уже работают по сети через PL/Perl). Поэтому отдельный «обзорный экран» дублировал бы то, что и так всегда на экране, — антипаттерн. Реальный пробел в другом: панели всего по 4 строки и не показывают production-DBA-сигналы, которые сегодня требуют обхода сайд-панелей (iostat/nicstat/fsstat) и экранов (databases/replication/bgwriter):

- насыщение по диску/сети/ФС (какое устройство — бутылочное горлышко прямо сейчас);
- агрегированная рабочая нагрузка (tps, ins/upd/del, temp);
- суммарный размер всех БД + рост + cache hit ratio;
- насыщение пула воркеров;
- репликация — лаг, слоты, retained WAL, бэклог архивации, senders/receivers;
- стоимость чекпоинтов.

Verbose-режим расширяет всегда-видимый заголовок до этих сигналов **по требованию**, над любым экраном, на котором работает DBA.

**Дифференциатор vs pg_activity:** (1) всё работает по сети через PL/Perl-схему `pgcenter`; (2) расширенный заголовок едет вместе с DBA по детальным экранам — у pg_activity богатый блок зафиксирован только над списком процессов.

## Пользовательские истории

- Как дежурный DBA, я хочу нажатием `v` развернуть верхние панели в расширенную сводку здоровья, чтобы за один взгляд понять, где болит, и действовать на экране ниже или перейти по хоткею.
- Как DBA, я хочу держать verbose включённым, оставаясь на детальном экране (например `tables`), чтобы health-контекст был перед глазами над таблицей, в которую я углубился.
- Как DBA, я хочу чтобы verbose-агрегаты были полностью консистентны с полными панелями (iostat/nicstat/fsstat) и экранами (databases/replication/bgwriter), чтобы одно и то же число не отличалось между обзором и детальным видом.
- Как DBA, я хочу видеть `n/a` вместо `0` для недоступных сигналов (выключен `archive_mode`, нет репликации, нет PL/Perl-схемы по сети, нет прав), чтобы не принять пропуск за реальный ноль.
- Как DBA на нагруженном инстансе, я хочу чтобы verbose не подсыпал лишней нагрузки, чтобы мониторинг не усугублял инцидент.

### Пользовательские сценарии

**Сценарий 1: триаж инцидента (расширенная сводка ловит то, что иначе собирали бы руками)**
1. В 03:00 алерт по лагу реплики. DBA подключается и нажимает `v` — панели разворачиваются.
2. Левая панель: `iostat` показывает устройство с max util 100% и высокий `w/s` — диск пишет на пределе. `filesyst` показывает 74% занятости ФС каталога данных.
3. Правая панель: `replication` показывает большой `lag` и растущий `archiving backlog`; `bgwr/ckpt` — чекпоинты идут как `req`, не `timed`.
4. Результат: один разворот заголовка указал на WAL/checkpoint-шторм + насыщение диска, не уходя с текущего экрана. DBA жмёт `o`/`b`, чтобы подтвердить детали. Без verbose пришлось бы под давлением обойти iostat → bgwriter → replslots вручную.

**Сценарий 2: контекст здоровья поверх детальной работы**
1. DBA расследует bloat на экране `tables` (`t`).
2. Включает `v` — над таблицей появляется расширенная сводка; таблица ужимается, но остаётся видимой.
3. DBA видит в `workload` всплеск `upd/s` и в `databases` рост размера — связывает bloat с нагрузкой, не покидая экран `tables`.

**Сценарий 3: консистентность с полными панелями**
1. DBA видит в verbose-строке `iostat` устройство с max util.
2. Открывает полную панель iostat (`B`) — цифры по этому устройству совпадают (один источник, verbose лишь округляет до целых).

## Дизайн и интерфейс

Интерфейс — CLI/TUI (gocui). Новых форм ввода нет. `v` — переключатель verbose on/off, персистентный между экранами.

### Раскладка (схематично; панели — левая/правая половины экрана)

**Левая панель `sysstat`** (4 существующие строки + 3 verbose):
```
 (loadavg / %cpu / MiB mem / MiB swap — как сейчас)
  iostat: 12 devices, 100% max util, 1135 rMB/s, 34152 r/s, 1546 wMB/s, 17852 w/s
 nicstat: 4 devices, 100% max util, 4345 rMbps, 6543 wMbps, 3451/0 err/coll
filesyst: /dev/nvme0n1p2 on / (ext4), 468.1G size, 346.4G used, 74% use
```

**Правая панель `pgstat`** (4 существующие строки + 5 verbose):
```
 (state / activity / autovacuum / statements — как сейчас)
    workload: 1432 tps, 4132 ins/s, 5421 upd/s, 4235 del/s, 2341 ret/s, 123 tmp/s, 4 others
   databases: 1435GB per 7 databases, 1435 growth/s, 99.99% cache hit ratio
     workers: 0/8 workers/max, 0/4 logical workers, 0/8 parallel workers
 replication: 453GB wal size, 123GB lag, 1/12GB slots/retain, 15GB archiving backlog, 2/1 send/recv
   bgwr/ckpt: 12/3 timed/req, 245/30ms write/sync, 4 maxwritten
```

### Состав и источники строк

**`iostat`** — one-row агрегат из вспомогательной iostat-вьюхи (диск); полная консистентность с панелью iostat. Выбираем устройство с **максимальной `%util`** и показываем его значения:
- `devices` — число устройств (резерв 2 цифры, 0–99);
- `max util` — максимум `%util` (резерв 3 цифры);
- `rMB/s`, `wMB/s` — для max-util-устройства (резерв 4 цифры, динамический суффикс — см. ниже);
- `r/s`, `w/s` — для max-util-устройства (резерв 5 цифр).

**`nicstat`** — one-row агрегат из nicstat-вьюхи (сеть); консистентно с панелью nicstat. Устройство с макс. `%util`:
- `devices` — общее число устройств;
- `max util` — максимум `%util` (резерв 3 цифры);
- `rMbps`, `wMbps` — для max-util-устройства (резерв 4 цифры, динамический суффикс);
- `err/coll` — композит `IErr+Oerr` и `Coll` для max-util-устройства, формат `1111/2222 err/coll` (резерв 4 цифры на каждое).

**`filesyst`** — one-row агрегат из fsstat-вьюхи; консистентно с панелью fsstat. Показываем **ФС каталога данных** (`data_directory`):
- `device` — физическое устройство (ссылка в `/dev`);
- `mounted` — точка монтирования (обрезка до 10 символов);
- `filesystem` — тип ФС;
- `size` / `used` / `use %` — полный размер / занято / занято в процентах.

**`workload`** — агрегат `pg_stat_database` (per-interval rate):
- `tps` — `xact_commit + xact_rollback` в секунду;
- `ins/s` / `upd/s` / `del/s` — `tup_inserted` / `tup_updated` / `tup_deleted` в секунду;
- `ret/s` — `tup_returned` в секунду;
- `tmp/s` — `temp_files` в секунду;
- `others` — `deadlocks + conflicts + checksum_failures` **за интервал** (значение, без `/s`).

**`databases`** — ключевая информация о размерах:
- `XGB per Y databases` — `sum(pg_database_size)` по всем БД и их количество;
- `growth/s` — изменение суммарного размера за секунду;
- `cache hit ratio` — **per-interval** `Δblks_hit / Δ(blks_hit + blks_read)` (кумулятивный был бы вечные 99.9%, не реагирующие на инцидент).

**`workers`** — насыщение пула воркеров; активные счётчики из `pg_stat_activity.backend_type`, лимиты из GUC:
- `workers/max` — **зонтик** `max_worker_processes`: занято слотов / всего (исчерпание блокирует и parallel, и logical, и extension-воркеры);
- `logical workers` — активные logical replication воркеры / `max_logical_replication_workers`;
- `parallel workers` — активные parallel воркеры / `max_parallel_workers`.

**`replication`** — данные репликации:
- `wal size` — размер каталога `pg_wal`;
- `lag` — общий лаг репликации в байтах (worst-case по standby);
- `slots/retain` — число слотов / максимальный retained WAL;
- `archiving backlog` — бэклог архивации WAL: **`count(*.ready в pg_wal/archive_status) × wal_segment_size`** (чистый SQL через `pg_ls_dir`, работает по сети; `n/a` при `archive_mode=off` или нехватке прав);
- `send/recv` — активные walsenders / walreceivers (из `backend_type`).

**`bgwr/ckpt`** — фоновая запись/чекпоинты (минимальный критичный набор, без буферов — они дублировали бы экран bgwriter):
- `timed/req` — плановые / форсированные чекпоинты, **абсолютные** кумулятивные (ratio и есть сигнал; ADR [004]);
- `write/sync ms` — стоимость чекпоинта (per-interval дельта);
- `maxwritten` — сколько раз bgwriter упёрся в `bgwriter_lru_maxpages`.

### UX-поведение
- **Toggle и персистентность.** `v` включает/выключает verbose; состояние сохраняется между переключениями экранов (как сайд-панели), так что сводка остаётся над текущей таблицей.
- **Консистентность.** Verbose-агрегаты вычисляются из тех же данных, что и полные панели/экраны (`Diskstats`/`Netdevs`/`Fsstats`, `pg_stat_database`, `pg_replication_slots`, `pg_stat_bgwriter`/`pg_stat_checkpointer`); verbose лишь округляет до целых.
- **Форматирование.** Значения целые, дробная часть округляется **вверх** (полные панели показывают дробные — для verbose это избыточно и дорого по символам). Резервирование разрядов: суффиксы и раскладка статичны, меняются только цифры (паттерн уже есть в sysstat/pgstat).
- **Динамический суффикс единиц.** При переполнении зарезервированных разрядов единица переключается (MB/s→GB/s, Mbps→Gbps), чтобы топовое железо (NVMe-массивы >9.7 GB/s, 25/40/100GbE) не ломало раскладку. Это **новый паттерн** в коде — вводится с нуля.
- **Деградация в `n/a`.** Недоступный сигнал (нет PL/Perl по сети, `archive_mode=off`, нет репликации, нет прав, версия PG без нужного view) → литерал `n/a`, не `0` и не пусто; каждый сигнал собирается независимо, падение одного не гасит остальные строки.
- **Первый тик.** Метрики-дельты на первом тике (нет prev) → `n/a` + временная подсказка `collecting...` в cmdline, гаснет после первого рефреша.
- **Guard и консистентность.** Throttle (разрежение опроса по латентности) применяется **только к дорогим агрегатам без живого панель-двойника** (размеры БД, growth). Системные строки (iostat/nicstat/filesyst), у которых есть полные панели для перекрёстной сверки, собираются **каждый тик** при verbose и не throttle-ятся — поэтому обещание консистентности с панелями держится. Throttle-нутый источник показывает последнее известное (stale) значение, не `n/a`, молчаливо, и сам восстанавливается.
- **Вертикальное место.** Verbose увеличивает верхнюю полосу, таблица `dbstat` ужимается (прецедент — панель `extra` делит место с `dbstat`). Минимальный порог: высота должна вмещать verbose-полосу + `cmdline` + строку-заголовок таблицы + ≥1 строку данных; иначе verbose не разворачивается и в cmdline показывается подсказка.
- **Ширина.** Verbose-строки длинные и рассчитаны на широкий терминал; на узких — усечение по краю панели (как и существующие панели/таблица). Verbose имеет смысл на широком терминале.

## Как должно работать

### Основной сценарий
1. Пользователь нажимает `v`.
2. Верхние панели разворачиваются: `sysstat` +3 строки, `pgstat` +5 строк; включается сбор verbose-источников; таблица `dbstat` ужимается.
3. На каждом тике (интервал — `z`) панели рендерят compact- и verbose-строки с актуальными значениями.
4. Пользователь оценивает сводку, действует на текущем экране или жмёт хоткей детального экрана. Повторное `v` сворачивает панели и останавливает verbose-сбор.

### Граничные случаи
- **Remote без PL/Perl-схемы** → системные verbose-строки (`iostat`/`nicstat`/`filesyst`) в `n/a`.
- **Remote / симлинк `data_directory`** → симлинк по сети не резолвится; `filesyst` показывает ФС по нерезолвленному пути (документированное ограничение).
- **`archive_mode=off`** → `archiving backlog` в `n/a`.
- **Нет репликации / standby** → `lag`/`slots`/`send/recv` пустые/`n/a`; WAL/checkpoint recovery-aware (checkpoint→restartpoint).
- **Версии PG** → `bgwr/ckpt` версионно-aware (PG 17+ `pg_stat_checkpointer`); `others`-колонки `pg_stat_database` доступны с поддерживаемых версий; недоступное → `n/a`.
- **Первый тик** → дельты `n/a` + `collecting...`.
- **Разрыв соединения / рестарт сервера** → существующий путь обработки ошибок/reconnect pgcenter; не частичный «битый» кадр.
- **Узкий/низкий терминал** → строки усекаются по краю; при нехватке высоты verbose не разворачивается.
- **Переполнение разрядов** → динамическое переключение суффикса (GB/s, Gbps).

## Критерии приёмки
- [ ] `v` переключает verbose-режим обеих панелей; повторное `v` возвращает compact; состояние персистентно между экранами.
- [ ] В verbose `sysstat` показывает 3 доп. строки (iostat/nicstat/filesyst), `pgstat` — 5 доп. строк (workload/databases/workers/replication/bgwr-ckpt); раскладка и суффиксы статичны (резерв разрядов), меняются только значения.
- [ ] Verbose-агрегаты консистентны с полными панелями/экранами (одно устройство/число не расходится); verbose показывает целые с округлением вверх.
- [ ] `iostat`/`nicstat` показывают устройство с максимальным `%util` и его r/w-значения; `filesyst` — ФС каталога данных.
- [ ] `workload` = per-interval rate из `pg_stat_database` (tps = commit+rollback; others = значение за интервал без `/s`); `databases.cache hit ratio` — per-interval, а не кумулятив.
- [ ] `workers/max` = насыщение зонтика `max_worker_processes`; logical/parallel — активные из `backend_type` vs GUC.
- [ ] `replication.archiving backlog` = `count(*.ready) × wal_segment_size`; `n/a` при `archive_mode=off`/нехватке прав; senders/receivers — активные из `backend_type`.
- [ ] `bgwr/ckpt` показывает timed/req (абсолютные), write/sync ms (дельта), maxwritten; версионно-aware для PG 17+.
- [ ] Недоступные сигналы → литерал `n/a` (никогда `0`/пусто); падение одного источника не гасит остальные verbose-строки.
- [ ] При переполнении зарезервированных разрядов единица переключается (MB/s→GB/s, Mbps→Gbps).
- [ ] Дорогой сбор идёт **только** когда verbose включён; защита от нагрузки — tiering ритмов + guard по латентности; throttle применяется только к дорогим агрегатам без панель-двойника (системные строки собираются каждый тик ради консистентности); throttle-нутый источник держит stale-значение, не `n/a`; вторая пользовательская ручка интервала не появляется.
- [ ] На первом тике дельты `n/a` + `collecting...` в cmdline, гаснет после первого рефреша.
- [ ] Verbose увеличивает верхнюю полосу (с пересчётом геометрии `layout()` и сдвигом `cmdline`), `dbstat` ужимается; при нехватке высоты (verbose-полоса + cmdline + заголовок + ≥1 строка данных) verbose не разворачивается и показывается подсказка.

## Ограничения
- **Версии PostgreSQL:** активная поддержка PG 14–18.
- **Стек:** Go 1.25+, gocui (TUI), pgx/v5.
- **Сбор всех трёх системных источников одновременно (R1).** Verbose показывает iostat+nicstat+filesyst разом, тогда как существующий `collectExtra` собирает disk/net/fs по очереди (взаимоисключающе). Нужна новая ветка «собрать все три, когда verbose включён», не ломающая существующие сайд-панели. Это инвазивная часть фичи.
- **Многоритмовый сбор (tiering).** Дорогие источники (sum размеров БД + growth → slow-cached; системные → per-tick при verbose; `.ready`-бэклог) требуют разных ритмов + guard по латентности; всё спрятано за единственной ручкой `z`.
- **filesyst — только ФС каталога данных;** WAL/tablespaces на соседних дисках не показываются (известное ограничение). Симлинк `data_directory` резолвится только локально; по сети — нерезолвленный путь.
- **Net-new GUC-чтения:** `max_worker_processes`/`max_logical_replication_workers`/`max_parallel_workers` и `wal_segment_size` сейчас не читаются в `SelectCommonProperties` — их нужно добавить.
- **Динамический суффикс единиц — новый паттерн,** в коде его ещё нет, вводится с нуля.
- **Ширина:** verbose рассчитан на широкий терминал; на узких строки усекаются по краю панели.
- **Путь рендеринга (принтеры — лёгкая часть):** расширяем существующие free-form `Fprintf`-принтеры `printSysstat`/`printPgstat`; нового view, ветки `printStat` по `view.Name` и рендера в `dbstat` не требуется. **Инвазивное ядро — геометрия `layout()`:** сейчас высота панелей зашита (`y1=4`), а `cmdline` сидит на `y=3..5` поверх верхней полосы. Verbose-разворот до 7/9 строк требует verbose-aware динамической геометрии: пересчёт высоты полосы, сдвиг `cmdline` вниз и ужатие `dbstat`. Это, а не принтеры, основная работа по рендерингу.
- **TUI-only:** verbose — режим отображения, в `record`/`report` не участвует.
- **Единственная ручка интервала:** `z`.

## Риски
- **Риск (R1 — сбор всех трёх системных источников):** новая always-on (при verbose) ветка disk+net+fs может задеть взаимоисключающую логику существующих сайд-панелей iostat/nicstat/fsstat. **Митигация:** независимая ветка сбора, гейтится флагом verbose; существующую mutual-exclusion не трогаем; верхнеуровневая консистентность — переиспользуем те же readers/структуры; регрессионно тестируем сайд-панели.
- **Риск (стоимость дорогих агрегатов на нагруженном инстансе):** sum размеров БД, агрегаты statements/databases, обход `archive_status`. **Митигация:** tiering ритмов (slow-cached для размеров) + guard по латентности; сбор только при verbose; `.ready`-бэклог дёшев (один `pg_ls_dir`).
- **Риск (новый паттерн динамического суффикса):** код форматирования с переключением единиц — новая поверхность для ошибок раскладки. **Митигация:** чистая функция форматирования + property/таблично-управляемые юнит-тесты на границах переполнения.
- **Риск (вертикальное место):** verbose съедает строки таблицы. **Митигация:** пользовательский toggle (включают для взгляда, выключают для глубокой работы) + height-guard, не разворачивающий verbose на низком терминале.
- **Риск (расширяемость шва сбора):** будущие сигналы не должны требовать переписывания. **Митигация:** collect-функция как реестр источников с пер-источниковым состоянием ритма на `Collector`; добавление источника — локальная регистрация (юнит-тест шва).
- **Риск (переделка геометрии `layout()`):** динамическая высота панелей + сдвиг `cmdline` — основное инвазивное изменение в общем UI-цикле, затрагивающее все экраны. **Митигация:** geometry — чистая функция от verbose-флага и размеров терминала (юнит-тест без gocui); табличный путь и compact-режим остаются без изменений; ручная QA на нескольких размерах терминала.

## Технические решения
- Мы решили **сделать обзор verbose-режимом верхних панелей, а не отдельным экраном**, потому что две панели pgcenter уже и есть мини-обзор — отдельный экран дублировал бы их (антипаттерн). Verbose расширяет существующие `printSysstat`/`printPgstat`: нет нового view, нет ветки `printStat`, нет рендера в `dbstat`. `v` — mode-toggle (идиома сайд-панелей `B`/`F`/`N`/`L`), персистентный между экранами.
- Мы решили **гейтить дорогой сбор флагом verbose** (механизм `CollectExtra`, ADR [001]) и **собирать все три системных источника одновременно** при verbose, потому что обзор показывает iostat+nicstat+filesyst разом; существующий по-очереди-`collectExtra` это не даёт. Собираем только при verbose, чтобы не нагружать инстанс зря.
- Мы решили **консистентность через переиспользование тех же данных**, что и полные панели/экраны, потому что одно и то же число не должно расходиться между обзором и детальным видом; verbose лишь округляет до целых вверх.
- Мы решили **cache hit ratio считать per-interval, а tps = commit+rollback**, потому что кумулятивный hit ratio не реагирует на инцидент, а откаты — тоже транзакции и сигнал.
- Мы решили **archiving backlog считать по `.ready`-файлам × `wal_segment_size`**, потому что это операционно и есть бэклог, и это чистый SQL (`pg_ls_dir`), работающий по сети без PL/Perl.
- Мы решили **workers/max сделать зонтиком `max_worker_processes`**, потому что его исчерпание блокирует все типы воркеров — самый сильный top-level сигнал; активные счётчики — из `backend_type`.
- Мы решили **ужать `bgwr/ckpt` до timed/req + write/sync + maxwritten** (без буферов), потому что полный набор дублировал бы экран bgwriter.
- Мы решили **ввести динамическое переключение суффикса единиц** (MB/s→GB/s, Mbps→Gbps), потому что фиксированные 4 разряда переполняются на топовом IO/сети; паттерна в коде нет — вводим с нуля.
- Мы решили **не строить генерик-планировщик, а заложить расширяемый шов** (реестр источников с пер-источниковым ритмом + guard по латентности), потому что нужен tiering под разную стоимость источников, но без второй пользовательской ручки.
- Мы решили **filesyst показывать только для ФС каталога данных** и **не резолвить симлинк по сети**, потому что это покрывает основной кейс заполнения диска данных, а WAL/tablespaces и удалённый realpath — отдельная история (известное ограничение).
- Мы решили (рекомендация для tech-spec по **последовательности реализации**) разбить L-фичу на отделимые инкременты: сначала (1) toggle `v` + verbose-aware геометрия `layout()` + (2) системные строки (iostat/nicstat/filesyst, переиспользуют готовые структуры), затем (3) postgres-агрегаты (workload/databases/workers/replication/bgwr-ckpt) и (4) tiering + guard. Так каркас (самая инвазивная часть) проверяется раньше дорогих агрегатов.

## Тестирование

**Unit-тесты:** делаются всегда, не обсуждаются. Покрытие: композиция verbose-строк в `bytes.Buffer` (по каждой панели); вычисление агрегатов (выбор устройства с max `%util`; sum размеров БД; группировка по `backend_type`; `count(*.ready)`; per-interval cache hit; tps=commit+rollback; others за интервал); динамическое переключение суффикса на границах переполнения (property/табличный тест); форматирование `n/a` (sentinel → литерал, отличимый от `0`); расчёт высоты верхней полосы (compact vs verbose, height-guard); расширяемость шва сбора (регистрация доп. источника со своим ритмом без правки механики) — всё без живого PG.

**Интеграционные тесты:** делаем — потому что фича добавляет ~8 новых агрегатных SQL-запросов с версионными ветвлениями (PG 14–18), и корректность их выполнения и деградации проверяется только против живого PostgreSQL. Покрытие: новые агрегаты по источникам, версионные различия (`pg_stat_checkpointer` на 17+), пути деградации (нет репликации, `archive_mode=off`, remote без схемы, нехватка прав).

**E2E тесты:** не делаем автоматически — в проекте нет TUI E2E-харнесса. Проверка ручная (см. ниже).

### Пользователь проверяет
- Включить `v` на широком терминале — обе панели разворачиваются, цифры читаемы; на узком — строки усечены по краю, на низком — verbose не разворачивается.
- Сверить verbose-агрегат с полной панелью (`B`/`N`/`F`) и экранами (`d`/`r`/`b`) — числа консистентны.
- Проверить remote с PL/Perl-схемой и без — системные verbose-строки показывают данные либо `n/a`.
- Проверить инстанс с `archive_mode=off` / без репликации / standby — соответствующие поля `n/a`/пусто, панель не падает.
- Понаблюдать первый тик (`collecting...` гаснет) и поведение при verbose, оставаясь на детальном экране (сводка над таблицей, таблица ужата).
